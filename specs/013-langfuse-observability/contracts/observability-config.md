# Contract: Observability Configuration & Compose Topology

This feature exposes no new HTTP endpoints on the AI service itself. Its interface to the rest of the system is (a) the environment-variable contract that turns tracing on/off, and (b) the set of new Compose services and the ports/hostnames other services (and developers) reach them through. Both are documented here as the "contract" a developer or operator needs, without duplicating research.md's rationale or plan.md's file layout.

## 1. Environment variable contract (`app/core/config.py`)

| Variable | Required? | Consumed by | Effect |
|---|---|---|---|
| `LANGFUSE_ENABLED` | No (default `true`) | `ai-service` (`app/core/config.py`, `app/core/observability.py`) | Explicit on/off switch, independent of the three below. `false` ⇒ tracing fully disabled, no instrumentation attempted, connection settings never required. |
| `LANGFUSE_HOST` | **Yes, when `LANGFUSE_ENABLED=true`** (fails startup otherwise) | `ai-service` | Base URL of the Langfuse instance. Defaults to the local self-hosted stack's address (`http://langfuse-web:3000`, see §2) — has no effect while it isn't reachable (fails open at request time, FR-005). |
| `LANGFUSE_PUBLIC_KEY` | **Yes, when `LANGFUSE_ENABLED=true`** | `ai-service` | Paired with secret key for OTLP Basic Auth to `{LANGFUSE_HOST}/api/public/otel`. Defaults to the same dummy value `langfuse-web`'s headless init seeds itself with (§2), so the local pair matches out of the box. |
| `LANGFUSE_SECRET_KEY` | **Yes, when `LANGFUSE_ENABLED=true`** | `ai-service` | See above. Must never be committed — `.env`-sourced, gitignored, same handling as `OPENAI_API_KEY`. |

**Contract guarantee**: with `LANGFUSE_ENABLED=true` (the default) and all three connection settings present, tracing is enabled for every LLM call the process makes on next startup, with no other code change required. `LANGFUSE_ENABLED=false` is a fully supported "tracing off" state, requiring none of the three connection settings — this is not treated as a misconfiguration and does not fail startup. `LANGFUSE_ENABLED=true` with any single connection setting empty, by contrast, **does** fail startup (`app/core/config.py`, research.md §11) — like `AI_SERVICE_TOKEN` or `STORAGE_S3_*`, this is treated as an oversight to catch immediately, not a silently-degraded runtime state, since the connection settings now default non-blank and can only end up empty via a deliberate-looking-but-incomplete override. `.env.example` ships `LANGFUSE_ENABLED=true` with host/keys already defaulted to match the local self-hosted stack (§2), so enabling that stack's `observability` compose profile is the only step needed to see traces — whether or not that profile is actually *running* has no bearing on startup (fails open at request time if there's nothing listening, per FR-005). Point host/keys at a cloud-hosted Langfuse instance instead, or set `LANGFUSE_ENABLED=false`, to change this.

**Auto-provisioning (local stack only)**: when the local stack is enabled (§2), `compose/langfuse/docker-compose.yml` passes `LANGFUSE_PUBLIC_KEY`/`LANGFUSE_SECRET_KEY` straight through as `LANGFUSE_INIT_PROJECT_PUBLIC_KEY`/`LANGFUSE_INIT_PROJECT_SECRET_KEY` on `langfuse-web`, alongside `LANGFUSE_INIT_ORG_ID`/`LANGFUSE_INIT_PROJECT_ID`/`LANGFUSE_INIT_USER_EMAIL`/`LANGFUSE_INIT_USER_PASSWORD`. This is Langfuse's own [headless initialization](https://langfuse.com/self-hosting/administration/headless-initialization): on first boot it seeds an org, project, admin user, and that exact API-key pair, so the value a developer puts in `LANGFUSE_PUBLIC_KEY`/`LANGFUSE_SECRET_KEY` is simultaneously what `ai-service` authenticates with and what the project is provisioned under — one pair, no manual UI signup (satisfies SC-003/US3-AS1 literally, not just "the containers start"). It's a one-shot seed: on later restarts, if the org/project already exist, `langfuse-web` skips re-creating them, so changing these values after first boot only takes effect against a fresh Postgres volume. `LANGFUSE_AUTH_DISABLE_SIGNUP` defaults to `true` since the account is always seeded — self-service signup is closed out of the box (FR-007). This auto-provisioning has no meaning against a cloud-hosted instance — there, `LANGFUSE_PUBLIC_KEY`/`SECRET_KEY` must be a real key pair generated in that instance's own UI, same as any other Langfuse client.

Langfuse's other first-run secrets (`NEXTAUTH_SECRET`, `ENCRYPTION_KEY`, `SALT`, Postgres/ClickHouse/Redis/MinIO credentials) are internal to `compose/langfuse/docker-compose.yml` and are not consumed by `ai-service` — they're a separate contract owned entirely by that sub-stack, generated once per environment and never shared with application code.

## 2. Compose service topology contract

New services added by `compose/langfuse/docker-compose.yml`, included into the top-level `compose/docker-compose.yml`. All six carry `profiles: ["observability"]` — **opt-in, off by default** — since they only make sense as a unit and Compose profiles are inherently per-service (no way to gate an `include:`d file as one thing):

| Service | Reachable from host? | Reachable from `ai-service`? | Purpose |
|---|---|---|---|
| `langfuse-web` | Yes — `localhost:3000` (UI + API + OTLP ingestion) | Yes — `http://langfuse-web:3000` | The URL developers open in a browser, and the value `LANGFUSE_HOST` should point to from inside the compose network. |
| `langfuse-worker` | No (internal only) | N/A (not called directly by `ai-service`) | Background job processing for Langfuse's own ingestion pipeline. |
| `postgres` (Langfuse's own) | No (internal only) | No | Langfuse's transactional metadata store. Distinct from this repo's own `postgres` service — no shared database, no shared credentials. |
| `clickhouse` | No (internal only) | No | Trace/observation analytics storage. |
| `redis` (Langfuse's own) | No (internal only) | No | Langfuse's job queue. Distinct from any Redis used elsewhere in the broader platform. |
| `minio` | No (internal only, except its own admin console if enabled) | No | S3-compatible blob storage for raw trace events/attachments. |

**Contract guarantee**: a plain `docker compose up` (or `make dev-up`) brings up `ai-service`/`mineru-server` only — none of the six Langfuse services start, and nothing else in the stack is affected (US3-AS2). Passing `--profile observability` (or `make dev-up-observability`) brings up all six alongside them with no separate command (US3-AS1) — satisfies spec SC-003 for the "enabled" case. This is what lets `ai-service` instead be pointed at a cloud-hosted Langfuse via `.env` alone, with zero local containers, and keeps unrelated compose invocations (`make prod-smoke`, which never passes the profile) genuinely free of unneeded services. Only `langfuse-web`'s port 3000 is published to the host; every other new service is reachable only from within the Compose network, consistent with this repo's existing internal-only convention for backing services (e.g. `seaweedfs`, `redis` in the backend deploy stack).

**Why not `COMPOSE_PROFILES=observability` in `.env.example`**: that's the standard Compose mechanism for "profile on by default, overridable" — the opposite of what's wanted here (off by default). Enabling observability is therefore always done via the explicit `--profile observability` CLI flag (or the `make dev-up-observability` target). See research.md §9 for a note on `compose/mineru/docker-compose.yml`'s own unrelated, self-referential use of `${COMPOSE_PROFILES:-}` — checked and confirmed not reachable from this project's service graph, so not a real collision risk today, just something to watch if that file's inclusion mechanism ever changes.

## 3. OTLP export contract (informational — third-party interface, not owned by this service)

`ai-service` exports spans to `POST {LANGFUSE_HOST}/api/public/otel/v1/traces`, HTTP Basic Auth with `base64("{LANGFUSE_PUBLIC_KEY}:{LANGFUSE_SECRET_KEY}")`. This is Langfuse's documented OTLP-over-HTTP ingestion contract (protobuf or JSON body); this service is a client of it, not an implementer.
