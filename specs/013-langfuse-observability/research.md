# Phase 0 Research: LLM Observability with Langfuse

## 1. Self-hosted service topology

**Decision**: Vendor a dedicated `compose/langfuse/docker-compose.yml` defining Langfuse v3's full six-service stack — `langfuse-web`, `langfuse-worker`, `postgres` (Langfuse's own, distinct from the AI service's `ai_appdb` instance), `clickhouse`, `redis`, and `minio` — then pull the whole file into the top-level `compose/docker-compose.yml` via the Compose spec's `include:` directive.

**Rationale**: Langfuse v3 (unlike v2, which only needed Postgres) requires Postgres for transactional metadata, ClickHouse for trace/observation analytics storage, Redis for its job queue, and S3-compatible blob storage for raw events and multi-modal attachments — six containers total, confirmed against the project's own reference `docker-compose.yml`. `include:` (Compose spec, stable since Compose v2.20) is designed for pulling a whole sub-stack's services into a parent project as a unit, which fits a 6-service bundle better than `extends:` — `extends:` merges one named service's config from another file and is what this repo already uses for the single-service `mineru-api` case, but repeating it six times here would lose the sub-stack's own inter-service `depends_on`/network wiring that the vendored file already defines correctly.

**Alternatives considered**:
- Reuse the backend deploy repo's shared Postgres/Redis/SeaweedFS — rejected per explicit decision to keep this repo's compose self-contained and independently runnable (see spec Assumptions).
- `extends:` per service (matching the `mineru-api` pattern) — rejected: six repeated `extends:` blocks are more error-prone than one `include:` and don't preserve the sub-stack's internal `depends_on` graph automatically.

**Operational notes carried into the plan**: ClickHouse and Postgres must run with `TZ=UTC` (non-UTC breaks Langfuse's trace queries); Redis needs `--maxmemory-policy noeviction` (eviction under memory pressure silently drops queued jobs); only `langfuse-web` (port 3000, UI/API) needs to be reachable from the host — worker, Postgres, ClickHouse, Redis, and MinIO's S3 port stay internal-only, consistent with this repo's existing "Pattern A" internal-only service convention already used for `seaweedfs`/`redis` elsewhere in the stack.

Source: [langfuse/langfuse `docker-compose.yml`](https://github.com/langfuse/langfuse/blob/main/docker-compose.yml), [Langfuse self-hosting overview](https://langfuse.com/self-hosting).

## 2. Application integration: auto-instrumentation

**Decision**: Use OpenTelemetry-native auto-instrumentation — `openinference-instrumentation-langchain`'s `LangChainInstrumentor().instrument()` — wired to an OTel `TracerProvider` with a `BatchSpanProcessor` exporting via `OTLPSpanExporter` to Langfuse's built-in OTLP ingestion endpoint (`{LANGFUSE_HOST}/api/public/otel`), authenticated with HTTP Basic Auth (`base64(public_key:secret_key)`) via the standard `OTEL_EXPORTER_OTLP_HEADERS` env var.

**Rationale**: Langfuse's own documented LangChain integration is the `langfuse.langchain.CallbackHandler`, which must be passed explicitly into every `.invoke()`/`.stream()` call's `config={"callbacks": [...]}` — i.e. manual, per-call-site wiring, not auto-instrumentation. `LangChainInstrumentor().instrument()` globally monkey-patches LangChain/LangGraph's internals for the whole process at startup, so every chain, graph node, and LLM call already in the codebase (`app/features/chat/*`, `app/features/ingestion/normalizer/*`, `app/features/plan/*`, etc.) is captured automatically with zero changes to `app/core/llm.py` or any call site — matching the requirement to use auto-instrumentation. Langfuse explicitly documents its OTLP endpoint as a supported ingestion path for any OTel-compliant instrumentor, OpenInference included, and this path composes cleanly alongside (or instead of) the native `CallbackHandler` if that's ever needed later.

**Alternatives considered**:
- Langfuse's native `CallbackHandler`, wired into `config={"callbacks": [...]}` at each top-level `.invoke()`/`.stream()` (chat turn, normalization run, plan generation, embedding calls) — reconsidered deliberately, not just dismissed: it drops the OTel/OpenInference dependency stack entirely and gets Langfuse's built-in `mask` hook for free, which would have let §3's redaction requirement be satisfied with a library primitive instead of a hand-rolled `SpanProcessor` (a genuinely better fit for Principle VIII). Still rejected: it only requires touching a small, bounded set of entry points *today*, but every *future* feature that adds a new LLM call must remember to wire the callback too — there is no structural backstop, unlike auto-instrumentation. Given this codebase's explicit trajectory of adding more Maestro sub-agents over time (Principle VI), that ongoing dependence on developer discipline is a real, compounding risk against SC-002 ("100% of LLM calls captured, no feature silently excluded"), and outweighs the dependency-footprint and native-masking advantages. Auto-instrumentation's zero-call-site-change property was the deciding factor.
- Langfuse's `@observe()` decorator — rejected: function-level, manual, same category as the CallbackHandler, without even its masking advantage.

Source: [Langfuse × LangChain integration](https://langfuse.com/integrations/frameworks/langchain), [Langfuse native OpenTelemetry docs](https://langfuse.com/integrations/native/opentelemetry), [Langfuse OTel/OpenInference cookbook](https://langfuse.com/guides/cookbook/otel_integration_arize).

## 3. Data minimization before export (Constitution Principle III, v2.3.0)

**Decision**: Pair the auto-instrumentor with a required (not optional) custom OTel `SpanProcessor` in `app/core/observability.py` that pattern-redacts span attribute *values* (account/card numbers, emails, phone numbers — the same PII shapes the codebase already recognizes) before the `BatchSpanProcessor` hands them to the exporter. `OPENINFERENCE_HIDE_*` whole-attribute toggles are explicitly rejected as the mechanism: hiding an entire `input.value`/`output.value` attribute would satisfy Principle III trivially but directly break FR-001 and the spec's own acceptance criteria ("...a trace appears... with inputs and outputs visible"). Redaction here must be pattern-level, preserving trace usefulness while stripping the PII shapes it actually needs to strip.

**Rationale**: As amended, Principle III exempts the *live LLM inference call* from redaction while the model backend is self-hosted — but that exception is explicit and narrow: it does not extend to "any secondary copy of that same content," and a Langfuse trace is named in the constitution text as exactly that kind of copy. So this feature carries the full, unconditional minimization requirement even though the LLM call it's observing does not. This is now unambiguous rather than the earlier open question of whether hiding everything or showing everything was the right default — it's neither; targeted pattern redaction is.

Implementation note: this should be a small, standalone helper owned by `app/core/observability.py` (or a shared `app/core/` redaction module logging.py can also draw from later), not an import from `app/features/chat/guards.py::strip_pii()` — that function is feature-slice code, and Principle V forbids a cross-cutting core module reaching into a feature slice. `strip_pii()`'s existing regex shapes (card/email/phone) are a reasonable starting pattern set to mirror, not a dependency to take.

**Alternatives considered**: Ship with no redaction and revisit later — rejected, conflicts with a NON-NEGOTIABLE principle rather than a soft guideline. `OPENINFERENCE_HIDE_*` as the sole mechanism — rejected: satisfies compliance by deleting the feature's value, which the amended constitution no longer requires accepting as a trade-off. Rely on Langfuse's own SDK-level masking config — rejected as the sole mechanism: that masking hook lives on the native `CallbackHandler`/SDK client, not on the pure-OTel auto-instrumentation path this feature uses.

Source: [Langfuse masking docs](https://langfuse.com/docs/observability/features/masking), [OpenInference masking/redaction guidance](https://arize.com/docs/ax/instrument/mask-and-redact-data).

## 4. Fail-open behavior (FR-005)

**Decision**: Wrap instrumentation bootstrap (`TracerProvider`/exporter construction and `LangChainInstrumentor().instrument()`) in `app/core/observability.py::configure()` in a broad try/except that logs via the existing core logger and lets startup continue on any failure. No code further wraps individual LLM calls — the OTel SDK's `BatchSpanProcessor` already exports asynchronously off the request path by design, so a slow/unreachable Langfuse endpoint delays or drops span export, not the request itself.

**Rationale**: Directly satisfies FR-005 ("tracing failures MUST NOT surface as user-facing errors or block responses") without adding any per-call-site defensive code, consistent with the whole point of choosing a global auto-instrumentation approach over manual wiring.

## 5. Testing approach

**Decision**: Unit-test `configure()`'s no-crash behavior (with and without Langfuse env vars present) and the redaction span processor's attribute-stripping logic directly, by constructing in-memory spans and asserting on their processed attributes. No test starts a real Langfuse/ClickHouse container or makes a real OTLP network call.

**Rationale**: Constitution Principle I already mandates mock-first, no-real-external-call tests for the LLM; this extends the same rule to the new external dependency this feature introduces. Langfuse's own containers are infrastructure this feature *starts*, not something the AI service's own test suite needs to validate — that's Langfuse's own project's responsibility.

## 6. Configuration surface

**Decision**: New optional settings in `app/core/config.py`, following the existing `Settings` pattern (e.g. `embedding_*`, `storage_s3_*`): `langfuse_host`, `langfuse_public_key`, `langfuse_secret_key`. All default to empty/disabled — `configure()` treats an empty host/key as "tracing disabled" and skips instrumentation entirely, rather than raising. This intentionally sits outside Principle VII's fail-fast requirement: that principle governs settings *required* for the service to run correctly, and observability is explicitly optional/best-effort per FR-005.

**Rationale**: Matches how `openai_api_key`/`embedding_api_key` already default to a mock placeholder rather than being force-required, and keeps local/CI runs working with zero Langfuse configuration.

Superseded by §10 — connection settings now default non-blank, and a dedicated `langfuse_enabled` switch was added, but the underlying "optional, outside Principle VII, never raises" framing here still holds.

## 10. Explicit `LANGFUSE_ENABLED` switch, decoupled from connection defaults

**Decision**: Add `langfuse_enabled: bool = True` to `Settings`. `configure()`'s gate becomes `if not settings.langfuse_enabled: return` (checked first), then a second check that all three connection settings are non-empty (now logs a warning if not, since — unlike before — reaching that point while still missing a value is more likely an oversight than a deliberate disable). `langfuse_host`/`langfuse_public_key`/`langfuse_secret_key` no longer default to `""`; they default to `http://langfuse-web:3000` and the same dummy key pair `langfuse-web`'s own `LANGFUSE_INIT_PROJECT_PUBLIC_KEY`/`SECRET_KEY` fallback in `compose/langfuse/docker-compose.yml` uses (§8) — the two files' fallbacks are written to match by construction, not by coincidence.

**Rationale**: §6's original design used "any connection setting blank" as the on/off signal, which conflated two independent questions: *should this process attempt tracing at all* and *where does it send traces if so*. That conflation had a real cost once §9 made the local Langfuse stack opt-in: an all-blank `.env` (the only way to keep the old "off" state) meant a developer enabling the `observability` compose profile still got nothing until they *also* separately typed in matching `LANGFUSE_HOST`/keys — two steps to get one outcome. Giving the connection settings sensible non-blank defaults (matching what the local stack seeds itself with) and moving the on/off decision to its own explicit flag collapses that to one step: enable the profile, get traces, nothing else to configure. When the profile isn't enabled, `ai-service` still tries to reach `http://langfuse-web:3000` and fails — silently, asynchronously, fail-open per FR-005/research.md §4 — so there's no behavioral cost to defaulting the attempt to "on."

**Alternatives considered**: Keep blank-as-disabled and just change what "blank" defaults to — rejected, doesn't actually add a switch, just moves which value means "off," and still requires the value to be blanked out to disable (worse ergonomics than a boolean). Default `langfuse_enabled=False`, requiring an explicit opt-in even with the profile running — rejected: reintroduces the exact two-step friction this change exists to remove, and there's no safety reason to require it given fail-open behavior already makes "on by default" harmless when nothing is listening.

**Carried into `compose/docker-compose.prod.yml`**: that file pins deliberately-broken values for `POSTGRES_HOST`/`STORAGE_S3_*`/etc. to keep `make prod-smoke` deterministic and dependency-free regardless of the developer's own `.env`; `LANGFUSE_ENABLED: "false"` was added there for the same reason — `langfuse_enabled` now defaulting to `true` would otherwise have that smoke test attempt a (fail-open, but still non-deterministic) network call.

## 11. Fail fast when enabled but misconfigured

**Decision**: Move the "enabled but a connection setting is empty" check out of `app/core/observability.py::configure()` (§10's warn-and-no-op) and into `app/core/config.py`'s existing module-level fail-fast block, alongside the `AI_SERVICE_TOKEN`/`STORAGE_S3_*`/`MINERU_API_URL` checks: `if settings.langfuse_enabled and _missing_langfuse_fields: raise RuntimeError(...)`. `configure()` keeps its own equivalent check too, but now purely as defense-in-depth — in normal operation, `app/core/config.py` is imported and validated well before `configure()` ever runs (`app/main.py`), so that branch should be unreachable in practice.

**Rationale**: §10 gave the connection settings non-blank defaults specifically so `LANGFUSE_ENABLED=true` (the default) plus the `observability` compose profile is enough to get tracing with zero further config. Given that, the *only* way to reach "enabled, but a connection setting is empty" is a developer deliberately blanking one of `LANGFUSE_HOST`/`PUBLIC_KEY`/`SECRET_KEY` in their own `.env` while leaving `LANGFUSE_ENABLED` at its default `true` — that's a config typo/oversight, not a supported "half-configured" state the way it was under §6's original blank-is-disabled design (where a blank value was the *only* way to express "off," so it had to be tolerated silently). Once there's a dedicated `LANGFUSE_ENABLED` flag for expressing "off" cleanly, an inconsistent enabled-but-incomplete combination is no longer something to tolerate quietly — it fits the same "required-when-a-condition-holds" shape as `MINERU_API_URL` (required unless `USE_MOCK_MINERU=1`) and `OPENAI_API_KEY`/`EMBEDDING_API_KEY` (required unless `USE_MOCK_LLM=1`), both already enforced by this same fail-fast block per Constitution Principle VII.

This is deliberately narrower than FR-005: FR-005 protects *requests* from a down-or-slow Langfuse *after* the service has started correctly — a live, reachable-but-currently-unavailable dependency. A blank connection setting while enabled is a *startup-time configuration defect*, not a runtime availability problem; Principle VII governs the former, FR-005 the latter, and they were being conflated under §6's original design. `configure()` itself must still never raise (FR-005 is unconditional once the process is running), which is why the check survives there too, just demoted to defense-in-depth against a state that should already be structurally unreachable.

**Alternatives considered**: Leave it as a runtime warning-and-no-op in `configure()` only (§10's original placement) — rejected per the reasoning above once `LANGFUSE_ENABLED` existed as a clean way to express "off"; a warning buried in structured logs is easy to miss compared to a startup failure, for a condition that's cheap to catch immediately and always indicates a mistake, never an intentional state.

## 7. Feature/flow attribution for usage metrics (US2, FR-006, SC-005)

**Decision**: Derive the "originating feature" span attribute from the request path rather than touching any individual LLM call site. `RequestLoggingMiddleware` (`app/core/request_logging.py`) already binds per-request context via `structlog.contextvars` before any feature code runs; extend it to also set a plain `contextvars.ContextVar[str]` (e.g. `current_feature`) to the second path segment after `/internal/` (`chat`, `ingestion`, `plan`, `embeddings`, `recommendations`, `analytics`, `transactions`) — the same value already used as each router's own `tags=[...]`. The redaction `SpanProcessor` from §3 reads this `ContextVar` and stamps it onto every span it processes as a Langfuse-recognized attribute before export, so it's available as a filter/group-by dimension in the Langfuse usage dashboard.

**Rationale**: Auto-instrumentation captures no per-feature label on its own — a span for `ChatOpenAI.ainvoke` looks identical regardless of which feature triggered it, since `get_chat_model()` is a single shared, `lru_cache`d constructor reused by chat, normalization, and plan alike. The alternative — passing `config={"tags": [...]}` at each `.ainvoke()`/`.astream()` call site — would touch every feature's LLM call individually, reintroducing exactly the per-feature wiring burden §2 rejected the `CallbackHandler` approach to avoid. Deriving the label from the request path via the existing per-request `ContextVar` pattern gets the same result with one small addition to a single, already-cross-cutting file, consistent with Principle VIII (reuse an existing pattern rather than invent a parallel one). This only labels HTTP-request-triggered LLM calls; any future background-job-triggered call (outside a request context) would need its own explicit attribution — out of scope here since no such call site exists yet.

**Alternatives considered**: Per-call-site `config={"tags": [...]}` — rejected, reintroduces the exact per-feature wiring burden this feature was designed to avoid. Rely on OpenInference's default span naming/class attributes alone — rejected: `ChatOpenAI`/`RunnableSequence` span names are identical across every feature and don't encode which feature triggered the call.

## 8. Eliminating the manual Langfuse signup step

**Decision**: Use Langfuse's own [headless initialization](https://langfuse.com/self-hosting/administration/headless-initialization) — a first-class, documented `LANGFUSE_INIT_*` env var set read by `langfuse-web` on startup — to auto-create the org, project, admin user, and API-key pair, instead of requiring a one-time manual browser signup. `LANGFUSE_INIT_PROJECT_PUBLIC_KEY`/`LANGFUSE_INIT_PROJECT_SECRET_KEY` reuse this repo's own `LANGFUSE_PUBLIC_KEY`/`LANGFUSE_SECRET_KEY` values directly, so the single key pair a developer sets is simultaneously what gets provisioned and what `ai-service` authenticates with. `LANGFUSE_AUTH_DISABLE_SIGNUP` now defaults to `true` (previously `false`) since an account always exists on boot.

**Rationale**: SC-003 and US3-AS1 in spec.md are written as "zero additional manual setup steps" / "without any manual account creation or external API key for the observability tool itself" — a literal reading that a bare `include:`d compose stack alone doesn't satisfy, since it still left project/key creation as a required manual browser step (originally captured honestly in quickstart.md as "out of scope" for this feature). Headless initialization is Langfuse's own supported mechanism for exactly this gap, not a workaround: it's documented, stable, and is the same pattern Langfuse recommends for scripted/production deploys. This was initially deferred as a possibly-undocumented risk; the official docs confirm it's a stable, first-party feature, so there's no longer a reason not to use it.

**Alternatives considered**: Scripting the signup via Langfuse's public HTTP API (register user → create org → create project → create API key) from an init container or entrypoint script — rejected: reimplements, in a brittle custom script, exactly what `LANGFUSE_INIT_*` already does natively and idempotently inside `langfuse-web` itself, with no extra container or startup ordering to get right. Leaving the manual step in place — rejected once it was confirmed the "risky/undocumented" concern was unfounded; the manual step was the only remaining gap against SC-003's literal wording.

**Caveat carried into the plan**: headless init is a one-shot seed — `langfuse-web` skips it if the org/project already exist, so changing `LANGFUSE_INIT_*`/`LANGFUSE_PUBLIC_KEY`/`LANGFUSE_SECRET_KEY` after the first successful boot has no effect against an existing `langfuse_postgres_data` volume; it only takes effect from a fresh volume.

Source: [Langfuse headless initialization](https://langfuse.com/self-hosting/administration/headless-initialization).

## 9. Making the local stack opt-in (compose profile), not default-on

**Decision**: Tag all six services in `compose/langfuse/docker-compose.yml` with the identical `profiles: ["observability"]`, so a plain `docker compose up` starts `ai-service`/`mineru-server` only — none of Langfuse's containers. Enabling requires `docker compose --profile observability up` (or `make dev-up-observability`, a new Makefile target). `ai-service`'s own `LANGFUSE_HOST`/`LANGFUSE_PUBLIC_KEY`/`LANGFUSE_SECRET_KEY` remain fully independent of this switch — set to a cloud-hosted Langfuse instance's real host/keys, tracing works with the local profile never enabled at all.

**Rationale**: §1's original `include:` decision made the local stack start unconditionally alongside everything else, on the reasoning that spec SC-003/FR-003 required zero-manual-setup self-hosting. That's still true, but it was an incomplete reading: FR-003 requires self-hosting *be available* with no external dependency, not that it *always run* — a developer who wants to point at a cloud-hosted Langfuse (or doesn't want tracing locally at all) shouldn't be forced to run six extra containers, including a ClickHouse/MinIO/Redis/Postgres set, just to start the rest of the stack. Spec.md's US3/FR-003/SC-003 were amended alongside this change to state the opt-in requirement explicitly. This also fixed a latent, unrelated bug: `make prod-smoke` (self-described as "boots it fully mocked with zero external dependencies") was, before this change, also unconditionally booting the entire Langfuse sub-stack, since it only overrides `ai-service` on top of the same base file.

Compose profiles are inherently per-service — there is no mechanism to gate an entire `include:`d file as a single unit — so all six services must carry the identical profile tag; they only function together (`langfuse-web`/`langfuse-worker` depend on all four backing services), so partial tagging would either break `depends_on` resolution or leave orphaned containers running when the profile is off.

**Alternatives considered**: `COMPOSE_PROFILES=observability` in `.env.example`, the standard Compose mechanism for "profile on by default, overridable" — rejected outright since the requirement here is off by default, not on by default; this mechanism doesn't fit regardless of the point below. (`compose/mineru/docker-compose.yml`'s `mineru-openai-server` service does read `${COMPOSE_PROFILES:-}` as one of its own profile tags — a self-referential trick so it piggybacks on whatever profile is active — but that service is never actually reachable from this project: `compose/docker-compose.yml` only pulls in `mineru-api` via `extends:`, which copies just that one named service, not its siblings from that file. Only `langfuse/docker-compose.yml` is pulled in wholesale, via `include:`. So `COMPOSE_PROFILES` would not, in fact, collide with anything here — verified directly with `docker compose config --services`. Noted only so a future change that starts `include:`-ing `mineru/docker-compose.yml` wholesale doesn't reintroduce this as a real hazard.) The explicit `--profile observability` CLI flag (via the `dev-up-observability` Makefile target) is used regardless, since it's the correct mechanism for "off unless asked for."

An override compose file (`docker-compose.cloud-langfuse.yml`) that redefines the six services with a profile that's never activated, layered in via an extra `-f` flag to *exclude* them — considered as an alternative to native profiles, rejected as more machinery for the same result profiles already provide natively.
