# Phase 0 Research: Consistent, Fault-Tolerant Environment Configuration

## 1. Removing compose-level redundancy (FR-001, FR-008)

**Decision**: Delete the `ai-service` service's `environment:` block from `compose/docker-compose.yml` entirely, leaving only `build: ..` + `env_file: ../.env`. Leave `compose/langfuse/docker-compose.yml` and `compose/mineru/docker-compose.yml` untouched (no `env_file` of their own — compose-level substitution is their only channel). Leave `compose/docker-compose.prod.yml`'s deterministic pins in place (just renamed, see §6) — they still take effect because Compose merges `environment:` across all `-f` files into one map before applying precedence, so `environment:` from *any* file in the merge outranks `env_file:` from the base file regardless of whether the base file also declares that key.

**Rationale**: `ai-service` is the only service in this repo's compose files with its own `env_file`. Its `environment:` block duplicated values already available through that file with `${VAR:-default}` substitution — and because Compose's precedence is `environment:` > `env_file:`, that duplication was not neutral: it was capable of silently shadowing a real `.env` override with a stale/generic literal whenever substitution didn't resolve to the same value (verified against the actual `.env` in this repo, which already diverges from the compose literal defaults for `OPENAI_BASE_URL`/`MODEL_NAME`).

**Alternatives considered**: Trim the block to only the values that currently differ from `Settings`' Python-side defaults (partial removal) — rejected once `docker-compose.prod.yml` was checked: it proved compose-level `environment:` is legitimately needed only for (a) services without their own `env_file`, or (b) an override file forcing determinism regardless of `.env` — neither reason applies to any of the base file's `ai-service` keys, so partial removal would have left redundant lines with no distinguishing rationale for which survived.

## 2. Configuration grouping mechanism (FR-004)

**Decision**: Reorganize `Settings(BaseSettings)` into nested `BaseModel` groups — `chat_model`, `embeddings`, `own_db`, `backend_db`, `storage`, `mineru`, `langfuse`, `logging` — with `model_config = SettingsConfigDict(env_nested_delimiter="__")` on the root `Settings`. `ai_service_token` stays a flat, ungrouped field on `Settings` (Principle VIII: a one-field group is structure without benefit). The chat/completion group is named `chat_model`, not `llm` — decided during task planning, since `llm` read as generic ("the LLM stack") when there are in fact two LLM-adjacent groups (`chat_model` and `embeddings`); `chat_model` names the specific thing it configures.

**Rationale**: `env_nested_delimiter` is pydantic-settings' native mechanism for exactly this shape — no hand-rolled prefix-parsing or custom `Field` aliasing scheme needed. Grouping by domain matches how the fields are already informally organized today (the existing `# ── own database ──` comment banners in `config.py` already delineate these exact boundaries) and how `docker-compose.prod.yml`'s override comments already talk about them ("the DB/storage overrides below").

**Alternatives considered**: Keep the flat `Settings` class and only add doc-comment groupings — rejected, doesn't get the testability win of §3 (validators need a natural per-group `self` to attach to) and doesn't fix the env var naming collisions across groups that already exist informally (e.g. `mineru_api_key` vs. a hypothetical future second service also wanting an `_api_key` field).

## 3. Fail-fast validation mechanism (FR-002, FR-003)

**Decision**: Replace the current module-level `if settings.x: raise RuntimeError(...)` blocks (which run once, at import time, against the process-wide `settings` singleton) with `@model_validator(mode="after")` on each group's `BaseModel` class, raising `ValueError` (which pydantic wraps into `ValidationError` on the enclosing `Settings()` construction). Checks that inherently span two groups (embeddings' required-ness is gated by `chat_model.use_mock`, not a separate embeddings-level mock flag) live on the root `Settings` model_validator instead, since only it has both groups in scope.

**Rationale**: Directly satisfies spec FR-003 — a test can now call `ChatModelSettings(use_mock=False, openai_api_key="__mock__")` (or `Settings(chat_model=...)`) and assert a `ValidationError`, per-construction, with no module reload or singleton mutation. The current design can only be exercised by reloading `app.core.config` with different `os.environ` state before import, which is why the existing test suite doesn't test these checks directly today.

**Alternatives considered**: `field_validator` per-field — rejected, several checks are inherently conditional on a sibling field within the same instance (e.g. `mineru.use_mock` gating `mineru.api_url`), which `field_validator` cannot see. A custom validation function called manually after `Settings()` (closer to today's shape, just moved into a function) — rejected, doesn't gain isolation from the singleton since it would still need to run once at import time against one instance to preserve today's "fails at boot" behavior; `model_validator` gets identical boot-time behavior for free (it runs automatically on every construction, including the one at import time) while also making per-instance construction in a test meaningful.

## 4. Closing the own-DB fail-fast gap (FR-002)

**Decision**: Change `postgres_password`'s Python-side default from the literal `"apppass"` to `""`, and add it (alongside `postgres_user`/`postgres_db`, whose current defaults `"appuser"`/`"appdb"` are equally fake-but-plausible) to the `own_db` group's fail-fast check, using the same "empty ⇒ missing" pattern already used for `storage`/`mineru`/`ai_service_token` — not a "known placeholder string" comparison (unlike `openai_api_key`'s `"__mock__"` sentinel, `own_db` has no single agreed sentinel value to check against, and inventing one is more fragile than removing the fake-default and reusing the existing empty-string convention).

**Rationale**: Every environment that actually needs `own_db` to work already supplies real values through `.env`/`env_file` (local dev) or compose `environment:` (deploy repo, per §6) — none of them depend on the Python-side literal default actually being usable. That default's only real effect today is letting a misconfigured environment boot silently against a database that doesn't exist with credentials nothing grants, exactly the failure mode Principle VII exists to prevent. Removing it and gating on emptiness is the smallest change that closes the gap without introducing a new validation concept.

**Alternatives considered**: Keep `"apppass"`-style defaults and add a denylist of "known-fake" values to check against — rejected as more fragile (a new fake value can be invented by a future developer copying the pattern) and inconsistent with how every other required-credential group in this codebase already signals "unset" (empty string), not "matches a blocklist."

## 5. `SecretStr` adoption and its call-site ripple (FR-006)

**Decision**: Type every credential-shaped field (`chat_model.openai_api_key`, `embeddings.api_key`, `own_db.postgres_password`, `backend_db.password`, `storage.s3_secret_key`, `mineru.api_key`, `langfuse.secret_key`, `ai_service_token`) as `pydantic.SecretStr`. Update every call site that currently interpolates one of these into a plain string, header, or comparison to call `.get_secret_value()` explicitly:

| File | Site | Today | After |
|---|---|---|---|
| `app/core/llm.py` | `ChatOpenAI(api_key=...)` | `SecretStr(settings.openai_api_key)` | `settings.chat_model.openai_api_key` (already `SecretStr` — the manual rewrap is now redundant and removed) |
| `app/core/embedding.py` | embeddings client construction | `SecretStr(settings.embedding_api_key)` | `settings.embeddings.api_key` (same simplification) |
| `app/core/observability.py` | OTLP Basic Auth header | `f"{...}:{settings.langfuse_secret_key}".encode()` | `f"{...}:{settings.langfuse.secret_key.get_secret_value()}".encode()` |
| `app/core/storage.py` | S3 client `aws_secret_access_key` | `settings.storage_s3_secret_key` | `settings.storage.s3_secret_key.get_secret_value()` |
| `app/core/security.py` | bearer token comparison | `credentials.credentials != settings.ai_service_token` | `credentials.credentials != settings.ai_service_token.get_secret_value()` |
| `app/features/chat/checkpointer.py` | Postgres DSN | `quote(str(settings.postgres_password), safe="")` | `quote(settings.own_db.postgres_password.get_secret_value(), safe="")` |
| `app/features/ingestion/mineru_client.py` | request header | `settings.mineru_api_key` | `settings.mineru.api_key.get_secret_value()` |
| `app/core/config.py` | `own_database_url`/`backend_database_url` properties | plain f-string interpolation | `.get_secret_value()` on the password field |

**Rationale**: `SecretStr.__str__()`/`__repr__()` deliberately render as a fixed masked placeholder (`**********`), which is exactly the protection FR-006 wants for logs/`repr()` — but the same masking means every one of the sites above, which today builds a real connection string, HTTP header, or does a real comparison, would silently start using the *masked placeholder text* instead of the real secret if `.get_secret_value()` is missed. This is a correctness bug, not just a lint nit — `checkpointer.py`'s `str(settings.postgres_password)` and `observability.py`'s f-string are the two sites most at risk, since both already coerce the field into a string implicitly rather than passing it as a typed value. Two of the eight sites (`llm.py`, `embedding.py`) already wrap the raw field in `SecretStr` manually before handing it to `langchain-openai` — confirming the library already expects `SecretStr` at that boundary — so those two call sites get *simpler*, not more complex.

**Alternatives considered**: Type only the fields not already touched by a "wrap in `SecretStr` at the call site" pattern (i.e. skip `chat_model.openai_api_key`/`embeddings.api_key` since they're "already handled") — rejected: leaving some credential fields as plain `str` while others are `SecretStr` reintroduces exactly the inconsistency FR-006 is meant to close (a `repr(settings)` would mask some secrets and not others, and a future field added to an un-typed group has no structural nudge toward masking).

## 6. Backend database becomes unconditionally required (FR-005, resolved clarification)

**Decision**: `backend_db.host`/`name`/`user`/`password` join the same "empty ⇒ fail startup" validator as every other required group (§4's mechanism), replacing today's silent `backend_database_url` → `None` fallback. `backend_database_url` becomes non-`Optional`; `app/backend_db/__init__.py::_ensure_engine()`'s `if url is None: raise` branch is removed since startup-time validation now guarantees a value exists by the time any code runs.

**Rationale**: Confirmed empirically, not just by design intent — `nbe-financial-advisor-backend/deploy/docker-compose.yml` (the actual production deployment, per `docker-compose.prod.yml`'s own comment) sets `BACKEND_DB_HOST`/`NAME`/`USER`/`PASSWORD` unconditionally for `ai-service`, with real defaults (`AI_READONLY_PASSWORD:-ai_readonly_pass`), not an empty/absent state. There is no real deployment where this is legitimately unset — the codebase's only justification for the optional path (`app/backend_db/__init__.py`'s docstring) was about unit tests/CI, and those don't need the config-layer optionality either: the tests that actually exercise backend-DB code paths already bypass `_ensure_engine()` via `monkeypatch.setattr("app.backend_db.get_backend_session", ...)`, so they never depend on `backend_database_url` resolving at all.

**Alternatives considered** (from the earlier clarification): conditionally required behind a feature flag (Option B) — rejected, no such flag exists or is needed once every real deployment sets it unconditionally; stays optional/lazy indefinitely (Option C) — rejected, directly contradicts the evidence in the deploy repo and weakens spec SC-002's "100% of required groups fail fast."

## 7. Cross-repo env var rename (constraint carried into Project Structure)

**Decision**: Rename env vars in lockstep across this repo's `.env.example`/`compose/docker-compose.prod.yml` AND `nbe-financial-advisor-backend/deploy/docker-compose.yml`/`.env.example`, using the new `GROUP__FIELD` names everywhere (e.g. `POSTGRES_DB` → `OWN_DB__POSTGRES_DB`, `BACKEND_DB_HOST` → `BACKEND_DB__HOST`, `AI_STORAGE_BUCKET`-style deploy-repo indirection vars stay as the deploy repo's own naming choice, only the *ai-service-facing* key they map to renames).

**Rationale**: `nbe-financial-advisor-backend/deploy/docker-compose.yml`'s `ai-service` service has no `env_file` — every value it passes is a literal `environment:` key matching today's flat names exactly (`POSTGRES_HOST`, `BACKEND_DB_HOST`, `STORAGE_S3_BUCKET`, `USE_MOCK_LLM`, `MINERU_API_URL`, etc., confirmed by reading that file). A rename that only touches this repo would build and pass CI here while silently breaking real deployment — `Settings()` would simply never see the renamed keys and fail its own new fail-fast checks (§3, §4) at prod startup instead of at build/review time. Treating this as one atomic, two-repo change (both files renamed together) turns that failure mode into a would-be-caught-in-review diff instead of a would-be-caught-in-prod outage.

**Alternatives considered**: Keep old flat names as `validation_alias`/`AliasChoices` fallbacks alongside the new grouped names, so both work — rejected per explicit direction: the whole point of this pass is one source of truth per setting, and a permanent dual-name alias is the same "two ways to set the same thing" problem this feature exists to remove, just moved from compose-vs-env-file into old-name-vs-new-name.

## 8. Naming adjustments to the group/field names (decided during task planning, one reverted post-implementation)

**Decision**: The chat/completion group is `chat_model`, not `llm` (env prefix `CHAT_MODEL__`, was `LLM__`) — this rename stands. Its mock flag, and `mineru`'s mock flag, were briefly renamed from `use_mock` to `mock_enabled` during the same task-planning pass (`CHAT_MODEL__MOCK_ENABLED`, `MINERU__MOCK_ENABLED`), then **reverted back to `use_mock`** (`CHAT_MODEL__USE_MOCK`, `MINERU__USE_MOCK`) after implementation had already landed — both mock flags are `use_mock` in the final, shipped code.

**Rationale (group rename, kept)**: `llm` was ambiguous once there are two LLM-adjacent groups in the same `Settings` (`chat_model` and `embeddings`) — `chat_model` names the specific thing that group configures.

**Rationale (field rename, reverted)**: `mock_enabled` was tried as a more natural-reading boolean-state name than `use_mock`, applied consistently to both `chat_model` and `mineru`. Reverted back to `use_mock` post-implementation at explicit request — no rationale beyond preference was given for the revert, and none was needed: it's a same-effort naming call within the feature that hadn't reached a real deployment yet, reverted before it had any external consumers to consider.

**Alternatives considered**: Remove MinerU's mock-mode capability entirely while renaming — considered and explicitly rejected during the original rename pass: this is a naming/organization change, not a behavior change to which features support offline/mocked operation; removing mock support for MinerU would drop CI's ability to run without a live MinerU instance (Constitution Principle I), which is out of scope here. This conclusion is unaffected by the later `mock_enabled`→`use_mock` revert.
