# Quickstart: Object Storage Infrastructure

Validation guide for the `app/core/storage.py` capability once implemented
(see `tasks.md`, generated by `/speckit-tasks`, for the implementation
steps themselves). This guide only covers proving the feature works
end-to-end — it does not duplicate contract or data-model details; see
[contracts/storage-module-interface.md](./contracts/storage-module-interface.md)
and [data-model.md](./data-model.md) for those.

## Prerequisites

- `uv sync` has installed the new `aioboto3` runtime dependency.
- A reachable S3-compatible endpoint. Two supported scenarios:
  1. **No real endpoint available** — the fail-fast and unit-test paths
     below still fully validate; the live round-trip scenario is skipped.
  2. **A real SeaweedFS (or other S3-compatible) instance** with a
     pre-provisioned bucket and known access/secret keys, reachable from
     wherever tests run.

## Scenario 1 — Fail-fast on incomplete configuration (FR-008, US3)

```bash
STORAGE_S3_BUCKET= STORAGE_S3_ACCESS_KEY= STORAGE_S3_SECRET_KEY= \
  uv run python -c "import app.main"
```

**Expected outcome**: import fails immediately with a `RuntimeError`
naming the missing storage configuration — the process never reaches a
state where it would serve a request.

## Scenario 2 — Key-traversal rejection (FR-007, SC-004)

```bash
uv run pytest tests/core/test_storage.py -v
```

**Expected outcome**: the traversal-rejection test passes, demonstrating
`validate_storage_key` rejects an absolute path and a `..`-escaping key
before any network call — this test requires no network/Docker and always
runs in CI.

## Scenario 3 — Real round-trip against a configured S3-compatible store (FR-001–FR-005, US1)

Point `.env` (or exported environment variables) at a real reachable
instance:

```bash
export STORAGE_S3_ENDPOINT_URL=http://<your-seaweedfs-host>:<port>
export STORAGE_S3_BUCKET=<pre-provisioned-bucket>
export STORAGE_S3_ACCESS_KEY=<access-key>
export STORAGE_S3_SECRET_KEY=<secret-key>

uv run pytest tests/integration/test_storage_s3.py -v
```

**Expected outcome**:
- With the variables unset: the test reports **skipped**, not failed —
  confirming default CI stays fully offline (Constitution Principle I).
- With the variables set to a real reachable instance: the test performs
  a real `put_object` / `get_object` / `head_object` / `delete_object`
  round-trip and passes, confirming:
  - a written blob reads back byte-identical (US1, Acceptance Scenario 1),
  - existence checks reflect writes and deletes correctly (US1, Acceptance
    Scenarios 2–3),
  - listing under a prefix returns exactly the expected keys (US1,
    Acceptance Scenario 4).

## Scenario 4 — Config-driven environment targeting (US2)

Repeat Scenario 3 against a **second** S3-compatible endpoint/bucket
(different environment) with no source-code changes — only the
`STORAGE_S3_*` environment values differ.

**Expected outcome**: the same test suite passes unchanged against the
second target, confirming the service is retargetable purely through
configuration (SC-002).

**Verified 2026-07-12**: ran `tests/integration/test_storage_s3.py`
unchanged against two independent, temporarily-provisioned SeaweedFS
instances (distinct ports, buckets, and access/secret keys) — both passed
with only `STORAGE_S3_*` environment values differing, zero source changes
between runs. Instances were provisioned solely for this verification and
torn down immediately after; this feature does not provision or manage
SeaweedFS itself (see plan.md/research.md §5).
